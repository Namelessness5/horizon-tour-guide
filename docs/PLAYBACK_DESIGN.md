# 播放交互设计（讨论纪要）

2026-07-14。这是一次设计讨论的结论，**还没有实现**。
实现之前请先读完——第 3 节会让内容重新生成一遍，所以要在大批量补地标**之前**定下来。

---

## 出发点：四个问题

1. 玩家再来时不知道自己在哪（随机抽到一条 trivia，没有上下文）
2. 玩家降速想多听，程序没反应
3. 长内容播到一半遇到新地点，需要打断 / 衔接 / 切回
4. 是不是该建个状态机

这四点指向同一个结构性问题：**内容现在是"一整块音频"，它应该是"一串能在边界处切开的片段"。**

---

## 1. 地点身份走视觉，不走音频 —— 已采纳

**别往音频里塞地名。** 你路过涩谷十次，前九次都知道自己在哪，第十次的"欢迎来到涩谷十字路口"
就是纯噪音。而音频时间是最稀缺的资源——只有 4 到 26 秒。

**做法**：进入地点时字幕区淡入一个小标签（游戏里那种地名 toast），停留期间常驻或几秒后淡出。

- 零音频成本、零时间占用
- **不播内容的时候它也在** —— 高速冲过去没时间讲，但你依然知道刚路过了哪

音频报幕（一句地名，1.5 秒，只在玩家可能没有上下文时前置）是备选，优先级低于视觉标签。

### 顺带：字幕宽度要收窄

现在是屏幕宽度的 70%（`SubtitleWindow.MaxWidth`），太宽了。**收到 1/3 左右。**

---

## 2. 播放历史 —— **只记这次停留**，不跨停留 ✅ 已实现

原本打算修 intro 的生命周期（播过就出池、离开够久允许重播）。**决定：不做跨停留的历史。**

**不记任何跨停留的历史**：不记去过哪、不记跳过什么、不给 intro 特殊待遇。
出去再进来 = 重新随机（可能抽到同一条，这没关系），内容永远从头播。

理由：历史逻辑会牵出一堆纠缠的规则——听了一半算不算听过？多久之后允许重播？
intro 什么时候该重放？——而**现在还没有足够的实机体验来判断哪条规则是对的**。
先把流程跑通，让要优化的地方保持清晰。

代码上直接删掉了 `PlaybackHistory`（留着不用的代码比没有更糟）。

### 但**这一次停留里**播过什么，是要记的

最初写成一个 `playedThisVisit` 布尔：一次进入播一条，播完就闭嘴。
**实机之前就能看出这条不对**——玩家停在金阁寺前面不走，说明他想听，
而电台讲完一篇就再也不吭声了。

现在是：讲完一篇，人还在地点里，**歇 `--dwell`（默认 30 秒）接着讲下一篇**，
全部讲完才安静。`playedThisVisit` 从布尔变成了 `HashSet<string>`，
传给 `ContentSelector.Select(..., exclude)`。

记 id 而不是只记"播了几篇"，是因为**随机会重复**：七八篇里连抽两次，
撞上同一篇的概率并不低，而同一次停留把一篇讲两遍是明显的 bug。

`--dwell` 和 `--silence` 是两个旋钮，别混：

| | 管什么 | 防什么 |
| --- | --- | --- |
| `--silence=20` | **任意两篇之间**的地板 | 穿过密集地点时被连珠炮灌解说 |
| `--dwell=30` | **同一地点里**的续播节奏 | 停下来的人被电台晾着 |

### 那条"听了一半不该算听过"的顾虑

原本担心：12 段的内容只播了 2 段就开走，会被标成"播过"永久划出池子——内容白白浪费。
**不记跨停留的历史，这个问题自动消失了。** 下次进来它照样能被抽中，而且从头播。

## 3. 内容片段化 —— 已采纳，这是分水岭

### 为什么不做 mid-clip resume

原方案是"记录播放时间点、支持从中间恢复"。**不做**，两个理由：

1. **从任意时间点恢复会切在半句话中间。** 要避免这个，你就得知道句子边界在哪——
   而一旦需要句子边界，你已经在做片段化了，不如直接做。
2. **恢复未必是好体验。** 打断了 30 秒之后接上"……于是涩谷成了连接横滨的节点"，
   玩家早忘了前半句在讲什么。恢复只在打断很短时才有意义，而打断很短的时候本来就不该打断。

**已确认：不做续播。** 被打断的内容直接丢掉，标记为"播过一部分"，之后优先给别的内容。

片段化之后想加 resume 随时能加 —— **但反过来不行**，所以先做片段化。

### 片段化怎么做

一条稿子不再是一个 wav，而是**一串 3-8 秒的片段（按句切）**，每段带实测时长。

**流程（已确认）**：LLM 先写完整的稿子 → **按句切分** → 每段单独合成。
不让 LLM 直接输出分段结构。

一个改动解决三件事：

| | |
| --- | --- |
| **动态长度** | 预算 12 秒放前 3 段，预算 30 秒放全部 8 段 |
| **内容成本降到 1/3** | **short / medium / long 三份稿子取消**，改成一份稿子按预算播前 N 段 |
| **干净打断** | 不在半句话里切；等当前片段放完（最多 4 秒）再切 |

### 稿子结构：总分，不是倒金字塔

**已确认用总分结构**（先总述、后分述），不用倒金字塔。

要求写进 prompt：

- **第一段是总述**，必须能独立成立——它就是原来的 `short`

  > 曾经担心"总述太长，高速冲过去放不出来"（实测第一段合成出来 7 秒，而 150 km/h 
  > 只有 4 秒预算）。**已决定不管**：冲过去了才读完也无所谓，多边形后续会画大一些。
  > 所以不给总述加硬性时长上限，避免把 prompt 搞复杂。
- 后面每一段是一个**独立完整的分述单元**，从任意一段之后停下来都不突兀
- 段与段之间**不准有跨段依赖**（不能出现"另外"、"而这又导致"这类连接词开头的段）

最后一条是关键。总分结构本身不保证可截断——如果"分"的部分互相咬着，截断照样断头。

### 选内容的顺序（回答"是不是先随机内容再判断能播多久"）

**不是。顺序是反的：**

```
1. 用「第一段的时长」当准入门槛过滤候选
      —— 连总述都放不下的内容，直接不考虑
2. 从活下来的候选里挑一条（intro 优先 / 其余随机）
3. 播放时按预算贪心地放段，放到预算用完或段用完为止
```

关键变化：候选过滤看的是**最短可播长度（第一段）**，而不是整条的总长。
现在的 `clip.Seconds <= budget` 要换成 `clip.FirstSegmentSeconds <= budget`。

### 一个白捡的好处：播放中持续重估预算

片段化之后，每放完一段都可以重新算一次预算。于是：

- **玩家降速** → 预算变大 → 多放几段
- **玩家加速** → 预算变小 → 提前收尾

这**自动实现了第 2 节的需求**，不需要单独的"降速检测"逻辑。

## 4. 抢占和状态机 —— 优先级已下调，见第 6 节

原本打算做的：显式状态机（Silent / Playing / Interrupting / Bridging）+ 抢占策略
（只允许"更具体"打断"更笼统"；同层不打断；快讲完了不打断；衔接句要占预算）。

**但排除 region 之后，打断场景基本不存在了**（见下一节）。这一整块先搁置。

真要做时，记住：**难点不是画状态框，是抢占策略**——什么时候该打断。
那是策略，该住在 `PlaybackPolicy` 里，和"还能待几秒"放一起。

## 5. 第一版不打 region —— 这个决定的连锁影响

**已确认：这一版只打 landmark。** 顶多一个很大的"东京"地理标签做测试。

这不是个小决定，它连锁改变了好几件事：

### 打断场景消失了

打断的真实触发场景是**嵌套**——正在讲"东京"，你拐进了"涩谷十字路口"。
没有 region 内容，就没有"笼统的内容正在播、遇到具体的地点"这回事。

而**同层级之间不打断**（从一个景观直接开进另一个景观，让当前这条说完，最多多说 4 秒）。

所以：**抢占、衔接词、状态机——第一版都不需要。**

> ⚠️ 但有个前提：**那个"东京"标签不能有内容。**
> 它如果只是个视觉标签（告诉你在东京），没问题。
> 它一旦有内容，region 打断 landmark 的场景立刻回来，上面这段话全部作废。

### 片段化的理由变了

打断场景没了，片段化的价值就只剩：

1. **动态长度**（预算够就多讲）
2. **内容成本降到 1/3**（不用写三份稿子）
3. **播放中重估预算**（降速就多讲）

**这三条仍然足够支撑做它。** 尤其是第 2 条——按旧结构做十个地标，之后要全部返工。

### 位置栈退化成一层

`LocationResolver` 的 parentId 补全逻辑基本闲置（除非算上东京那个标签）。
代码不用改，留着，region 迟早会回来。

## 待定的问题

### 片段怎么合成？两条路，还没定

**(a) 每段单独 TTS**（推荐先试）

- 简单可靠，每段是独立 wav，播放器顺序播就行
- 质检重试按段做，粒度更细
- 段间有自然停顿，反而像电台的"说完一句停一下"
- **风险**：每段独立生成，语调会重置，连着听可能割裂

**(b) 整条 TTS，再按时间戳切**

- 韵律连贯自然
- 需要强制对齐找切点（whisper 能给词级时间戳）
- 更复杂

先试 (a)，听着割裂再上 (b)。

### 其他

- intro 允许重播的"足够久"是多久？先拍个几分钟，实机调
- 片段的目标时长（3-8 秒？）要看切出来的句子实际有多长
- 那个"东京"标签到底给不给内容？**给了的话，第 5 节的结论全部作废**
